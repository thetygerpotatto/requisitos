# Documento de Arquitectura del Sistema de Gestión de Órdenes y Entregas

## 1. Introducción

Este documento describe la arquitectura inicial del sistema de gestión de eventos y boletería para **VivaEventos S.A.S.**, una plataforma que permite a organizadores publicar eventos, gestionar la venta de boletas digitales, procesar pagos y controlar el acceso en puerta mediante validación de códigos QR. El sistema busca reemplazar procesos manuales (WhatsApp, Excel, consignaciones) con una solución backend robusta, orientada a microservicios, capaz de soportar picos de demanda, integrarse con pasarelas de pago reales y ofrecer trazabilidad completa de cada transacción.

## 2. Requisitos Funcionales

Los requisitos funcionales se presentan en forma de **historias de usuario**, especificando los **criterios de aceptación** y su evaluación bajo el criterio **INVEST**.

### Historias de Usuario

| **ID** | **Historia de Usuario** | **Criterios de Aceptación** | **INVEST** |
|--------|--------------------------|------------------------------|------------|
| **US-01** | Como **organizador**, quiero publicar un evento con nombre, descripción, fecha, lugar, tipos de boleta (general, VIP, estudiante), precio por tipo y cupo máximo por tipo, para que los compradores puedan encontrarlo y adquirir boletas. | 1. El sistema valida que todos los campos obligatorios estén presentes (nombre, fecha, lugar, al menos un tipo de boleta con precio y cupo). 2. El evento queda en estado "borrador" hasta que el organizador lo active. 3. Una vez activo, el evento es visible para los compradores. 4. El sistema registra quién creó el evento y cuándo. | **Cumple.** Es independiente, negociable, valiosa para el negocio, estimable, pequeña y verificable con los criterios definidos. |
| **US-02** | Como **comprador**, quiero buscar eventos disponibles por nombre, ciudad o fecha, y ver el detalle de un evento (descripción, tipos de boleta disponibles, precios y cupos restantes), para poder decidir si compro. | 1. La búsqueda retorna eventos activos que coincidan con al menos uno de los filtros aplicados. 2. El detalle del evento muestra en tiempo real los cupos restantes por tipo de boleta. 3. Eventos con cupo agotado en todos sus tipos se muestran como "agotado" sin permitir compra. | **Cumple.** Independiente del flujo de compra, verificable, de tamaño adecuado para una iteración. |
| **US-03** | Como **comprador**, quiero seleccionar uno o más tipos de boleta, aplicar opcionalmente un código de descuento válido y proceder al pago, para adquirir mis boletas de forma segura. | 1. El sistema reserva temporalmente el cupo seleccionado durante el proceso de pago (máximo [definir tiempo] minutos). 2. Si el código de descuento es válido y está activo, el sistema aplica el porcentaje o valor de descuento correspondiente al total. 3. Si el código es inválido o está vencido, el sistema informa el error sin bloquear la compra. 4. El sistema no permite confirmar una compra que exceda el cupo disponible. | **Cumple.** Verificable y de valor claro. La reserva temporal la hace independiente del procesamiento del pago. |
| **US-04** | Como **comprador**, quiero pagar mi orden a través de una pasarela de pagos integrada ([herramienta: ej. Wompi, PayU, MercadoPago en modo sandbox]) y recibir la confirmación del estado de mi pago (aprobado, pendiente o rechazado), para saber si mi compra fue exitosa. | 1. El sistema redirige o muestra el formulario de pago de la pasarela seleccionada. 2. Ante un pago aprobado, la orden pasa a estado "pagada" y se generan las boletas digitales. 3. Ante un pago pendiente, la orden queda en estado "pendiente" hasta recibir la notificación (webhook/callback) de la pasarela. 4. Ante un pago rechazado, la reserva de cupo se libera y la orden queda en estado "fallida". 5. El sistema nunca genera boletas para órdenes no confirmadas. 6. El sistema no cobra dos veces por la misma orden (idempotencia en callbacks). | **Cumple.** Crítica para el negocio, independiente del flujo de validación en puerta, verificable con escenarios de sandbox. |
| **US-05** | Como **comprador**, quiero recibir una notificación (correo electrónico y/o mensaje) con mis boletas digitales en formato QR una vez confirmado el pago, y un recordatorio previo al evento, para poder presentarlas en la entrada. | 1. Dentro de los [definir tiempo] minutos posteriores a la confirmación del pago, el sistema envía al comprador un mensaje con las boletas en formato QR, usando [herramienta: ej. SendGrid, AWS SES, Twilio]. 2. El sistema envía un recordatorio automático [definir: ej. 24 horas] antes del inicio del evento. 3. Si el envío falla en el primer intento, el sistema reintenta automáticamente y deja trazabilidad del estado del envío (pendiente, enviado, fallido). 4. Un fallo en el servicio de notificaciones no interrumpe ni revierte la confirmación de la compra. | **Cumple.** Desacoplada del flujo de pago mediante colas asíncronas, verificable, valiosa para el usuario final. |
| **US-06** | Como **personal de logística**, quiero escanear el código QR de una boleta en la entrada del evento con un dispositivo móvil, para validar que la boleta es auténtica, pertenece al evento correcto y no ha sido usada previamente. | 1. El sistema responde a la validación en menos de [definir tiempo] segundos bajo condiciones de red regular. 2. Si la boleta es válida y no ha sido usada, el sistema la marca como "utilizada" y devuelve confirmación de acceso permitido. 3. Si la boleta ya fue usada, el sistema devuelve un rechazo indicando uso previo, sin permitir el ingreso. 4. Si la boleta no corresponde al evento activo, el sistema devuelve un rechazo con el motivo. 5. El sistema garantiza que una boleta no pueda ser validada exitosamente más de una vez, incluso ante solicitudes concurrentes. | **Cumple.** Independiente, de alto valor operativo, verificable en condiciones controladas de red. |
| **US-07** | Como **organizador**, quiero modificar el precio, el cupo disponible o el estado de códigos promocionales de mi evento en curso, y que el sistema registre un historial de cada cambio (quién, qué, cuándo), para mantener control y trazabilidad ante reclamos. | 1. El organizador puede actualizar precio, cupo y estado de códigos promocionales de un evento que aún no haya ocurrido. 2. Cada modificación queda registrada en un log de auditoría con: ID del organizador, campo modificado, valor anterior, valor nuevo y timestamp. 3. No es posible reducir el cupo por debajo del número de boletas ya vendidas. 4. Los cambios de precio no afectan órdenes ya confirmadas. | **Cumple.** Aislada de otros flujos, verificable con los criterios de auditoría, tamaño adecuado. |
| **US-08** | Como **organizador**, quiero cancelar un evento y que el sistema notifique a todos los compradores sobre la cancelación, con indicación del proceso a seguir según la política de devolución definida, para gestionar la situación de forma ordenada. | 1. Al cancelar un evento, el sistema cambia su estado a "cancelado" y no permite nuevas compras. 2. El sistema envía notificación masiva a todos los compradores con boletas activas para ese evento, indicando el motivo (si se provee) y la política de devolución. 3. El sistema registra qué organizador realizó la cancelación y en qué momento. 4. La lógica de devolución se delega a la pasarela de pagos según sus capacidades; el sistema registra el resultado del intento de devolución por cada orden. | **Cumple.** Caso de negocio claro, verificable, independiente de la validación en puerta. |
| **US-09** | Como **administrador de VivaEventos**, quiero visualizar un panel en tiempo real con métricas clave por evento (boletas vendidas, ingresos acumulados, tasa de conversión del carrito, incidencias en pagos e ingresos), para tomar decisiones operativas durante y después del evento. | 1. El panel muestra en tiempo real (latencia máxima de [definir tiempo] segundos) la cantidad de boletas vendidas por tipo. 2. Se muestran las horas de mayor volumen de ventas. 3. Se registran y listan las órdenes con pagos fallidos o pendientes. 4. Se registran y listan los intentos de validación rechazados en la entrada. 5. Los datos son accesibles solo para usuarios con rol "administrador". | **Cumple.** Valiosa para la operación, verificable con los criterios de latencia y acceso por roles. |

## 3. Requisitos de Calidad

Los requisitos de calidad se presentan en forma de **historias de calidad**, siguiendo la estructura de Len Bass (Fuente → Estímulo → Artefacto → Entorno → Respuesta → Medida de Respuesta).

### Historias de Calidad

| **ID** | **Fuente** | **Estímulo** | **Artefactos** | **Entorno** | **Respuesta** | **Medida de Respuesta** |
|--------|------------|--------------|----------------|-------------|---------------|--------------------------|
| **RQ-01** | Múltiples compradores concurrentes | Apertura de venta de un evento popular: miles de solicitudes de compra simultáneas en los primeros minutos | Servicio de órdenes, servicio de inventario/cupos, base de datos de cupos | Operación normal bajo pico de demanda | El sistema procesa las solicitudes sin caídas, aplica control de concurrencia para no sobrevender y devuelve respuesta a cada comprador | El sistema soporta al menos [definir: ej. 1.000] solicitudes de compra concurrentes sin superar [definir: ej. 3] segundos de tiempo de respuesta en el percentil 95, y sin vender más boletas que el cupo disponible |
| **RQ-02** | Comprador | Intento de pago cuyo resultado queda en estado "pendiente" desde la pasarela de pagos | Servicio de pagos, manejador de webhooks/callbacks, servicio de órdenes | Operación normal con latencia o indisponibilidad temporal de la pasarela | El sistema mantiene la orden en estado "pendiente", espera la notificación de la pasarela (webhook) y actualiza el estado sin duplicar el cobro ni generar boletas prematuras | Cero órdenes cobradas dos veces. Cero boletas generadas antes de confirmación de pago. Tiempo máximo de procesamiento del callback: [definir: ej. 30] segundos tras recibirlo |
| **RQ-03** | Personal de logística en puerta | Escaneo de QR de una boleta durante el acceso masivo al evento, bajo condiciones de red móvil congestionada | Servicio de validación de boletas, base de datos de boletas | Evento en curso, red con latencia alta o intermitente | El sistema valida la boleta y responde con aprobación o rechazo de manera eficiente; garantiza que la misma boleta no puede validarse dos veces ante peticiones concurrentes | Tiempo de respuesta de validación ≤ [definir: ej. 2] segundos en el percentil 95 bajo carga normal. Tasa de doble validación exitosa = 0 (garantía de idempotencia) |
| **RQ-04** | Operador de infraestructura / equipo de soporte | El servicio de notificaciones se cae o experimenta errores durante el flujo de compra | Servicio de notificaciones, cola de mensajes ([herramienta: ej. Las notificaciones pendientes se encolan y se procesan cuando el servicio se recupera | El flujo de compra tiene disponibilidad ≥ 99.9% independiente del estado del servicio de notificaciones. | **RQ-05** | Atacante externo / auditor de seguridad | Intento de acceso a endpoints de organizador o administrador usando credenciales de un usuario con rol "comprador" | API Gateway, servicio de autenticación y autorización, todos los microservicios expuestos | Operación normal en producción | El sistema rechaza la solicitud con un error de autorización (HTTP 403), no expone información sensible y registra el intento en los logs de auditoría | 100% de las solicitudes no autorizadas son rechazadas. Cero exposición de datos de otros usuarios o de credenciales internas. Registro de cada intento fallido con timestamp, IP y endpoint en los logs |
| **RQ-06** | Equipo de soporte / administrador | Un comprador reporta que no recibió su boleta o fue cobrado dos veces; soporte necesita reconstruir el historial completo de la transacción | Sistema de trazabilidad / logs centralizados, todos los microservicios | Operación post-evento o durante incidente | El sistema permite reconstruir la secuencia completa de eventos de una orden: creación, reserva de cupo, intento de pago, respuesta de la pasarela, generación de boleta, envío de notificación | Dado el ID de una orden, soporte puede obtener en ≤ [definir: ej. 2] minutos la traza completa de eventos con timestamps y estados. ## 4. Restricciones del Sistema

Las restricciones establecen **limitaciones** en la arquitectura del sistema, ya sean tecnológicas, de negocio, regulatorias o de infraestructura.

### Lista de Restricciones

| **Tipo de Restricción** | **Descripción** |
|--------------------------|-----------------|
| **Tecnológica** | El sistema debe implementarse obligatoriamente como **arquitectura de microservicios**. Cada servicio debe tener responsabilidades claras y no puede acceder directamente a la base de datos de otro servicio. La comunicación entre servicios debe realizarse mediante APIs bien definidas o mediante mensajería asíncrona. |
| **Tecnológica** | El sistema debe desarrollarse únicamente en **[herramienta: definir lenguaje/framework backend, ej. Node.js + Express, Java + Spring Boot, Python + FastAPI]**. Este punto debe ser acordado por el equipo antes del inicio del desarrollo. |
| **Tecnológica** | La integración con la pasarela de pagos debe realizarse con una **pasarela real en modo sandbox** (no simulaciones ficticias). La pasarela seleccionada debe soportar: webhooks/callbacks para estados intermedios, flujo de pago con estados (aprobado, pendiente, rechazado) y pruebas documentadas. Candidatos: **[herramienta: ej. Wompi sandbox, PayU sandbox, MercadoPago sandbox]**. |
| **Tecnológica** | El servicio de notificaciones debe usar un **proveedor externo real** de envío de correos y/o mensajes. Candidatos: **[herramienta: ej. SendGrid, AWS SES, Twilio]**. El servicio de notificaciones debe estar desacoplado del flujo principal de compra mediante una cola de mensajes. |
| **Tecnológica** | El sistema debe ser **contenerizado con Docker** y tener un archivo `docker-compose` (u orquestación equivalente) que permita levantar el entorno completo de forma repetible, sin configuración manual. El repositorio debe incluir documentación de cómo levantar, probar y verificar el flujo de compra y validación. |
| **Tecnológica** | No se permite el almacenamiento de **secretos, llaves de API o tokens** en el código fuente ni en el repositorio. Deben gestionarse mediante variables de entorno o un gestor de secretos **[herramienta: ej. .env + docker secrets, AWS Secrets Manager, HashiCorp Vault]**. |
| **Tecnológica** | La base de datos principal del sistema debe ser **[herramienta: definir motor, ej. PostgreSQL, MySQL, MongoDB]**. Cada microservicio que requiera persistencia debe tener su propia base de datos o esquema aislado (patrón Database per Service). |
| **Tecnológica** | La cola de mensajes para comunicación asíncrona entre servicios (especialmente notificaciones) debe implementarse con **[herramienta: ej. RabbitMQ, Apache Kafka, AWS SQS]**. |
| **De negocio** | El MVP debe estar **entregable en 13 semanas**, con entregas parciales funcionales. No se construirán: motor de recomendaciones con IA, aplicación móvil nativa ni módulo de contabilidad completa. El alcance está limitado al backend del sistema descrito. |
| **De negocio** | El sistema debe soportar la operación en **al menos dos ciudades** desde el lanzamiento inicial, con capacidad de escalar a más ciudades sin rediseño de la arquitectura. |
| **De negocio** | El organizador es responsable de definir la **política de devolución** de su evento. El sistema facilita el proceso de devolución delegando la operación a la pasarela de pagos según sus capacidades; no implementa lógica de devolución manual. |
| **Regulatoria** | El sistema maneja **datos personales de compradores** (nombre, correo, datos de pago). Debe cumplir con los principios de la **Ley 1581 de 2012 (Habeas Data, Colombia)**: recolección con finalidad explícita, no uso para fines distintos, posibilidad de consulta y corrección por parte del titular. |
| **Regulatoria** | Los datos de tarjetas de crédito/débito **no deben ser almacenados** en los servidores del sistema. El procesamiento de datos sensibles de pago debe delegarse completamente a la pasarela de pagos certificada (PCI DSS). |
| **Regulatoria** | Toda acción administrativa (modificación de eventos, cancelaciones, cambios de precios, activación/desactivación de códigos) debe quedar registrada en un **log de auditoría inmutable** con identificación del actor, timestamp y detalle del cambio, para cumplir con requerimientos de trazabilidad ante reclamos. |
| **De infraestructura** | El entorno de despliegue inicial es **[herramienta: definir, ej. servidor propio, AWS, GCP, DigitalOcean]**. La arquitectura debe poder funcionar en un entorno de recursos moderados durante el MVP, sin requerir infraestructura de gran escala hasta que se justifique por crecimiento real. |
| **De infraestructura** | El servicio de validación de QR debe estar optimizado para funcionar eficientemente bajo condiciones de **red móvil con latencia alta o intermitente**, dado que el personal de logística opera en campo con señal variable. |
